Implementation Phase Presentation Script (Max 4 Minutes)

Hi everyone. In this part, I will explain how we implemented our gesture-based game, what challenges we ran into, and how we solved them.

Our project uses a TypeScript + Phaser game architecture, with Vite for development setup. We organized the game into scenes, so each part of the experience has a clear responsibility. For example, `BootScene` preloads all assets, `GameScene` handles the actual gameplay loop, `SettingsScene` manages tuning options, and `PauseScene` handles pausing and resume behavior.

During implementation, one of our main goals was to keep the code modular so we could improve one feature without breaking others. In `GameScene`, we used a state-driven design for game flow, with phases like initializing, waiting for next shot, ball in flight, hit, miss, and round over. This prevented messy conditional logic and made it easier to reason about what should happen in each moment.

Another major implementation decision was our input system. We wanted both keyboard input and webcam hand tracking, especially because webcam systems can fail depending on hardware permissions or lighting. To solve this, we created separate controllers: one for keyboard and one for MediaPipe hand tracking, then combined them through a shared input interface. That way, if webcam setup fails, the game still works with keyboard controls instead of crashing.

One of the biggest technical challenges was webcam noise and timing inconsistency. Raw hand landmark data is often jittery, and camera processing introduces latency. At first, this caused unfair misses and repeated accidental triggers.

We solved this in multiple layers:
- We average multiple hand landmarks to get a more stable hand center.
- We map hand position into three clear lanes (left, center, right) using tuned thresholds.
- We use a motion window for swipe detection and ignore tiny movements with a jitter deadzone.
- We apply a cooldown so one held pose cannot trigger many hits.
- We added early buffering and late grace windows, so webcam actions that are slightly early or slightly late can still count if they match gameplay timing.

This combination made interaction feel much more responsive and fair.

Another challenge was game balance. If the game stayed static, it became boring; if it scaled too fast, it became frustrating. To solve this, we implemented a dedicated `DifficultyManager`. It increases challenge only after a hit streak, by speeding up shots and reducing launch delay. On a miss, it immediately eases difficulty so the player can recover. We also clamp all tuning values to safe ranges to prevent invalid settings from breaking progression.

We also built a settings workflow to support experimentation and usability. Through `SettingsScene`, players can pick difficulty presets, adjust growth speed with a slider, toggle endless mode, and toggle music. These settings are stored in Phaser registry keys so different scenes stay synchronized. This was important because both UI scenes and gameplay scenes need shared, consistent state.

From a presentation and playtesting perspective, we added practical quality-of-life features: a pause menu that reflects current settings, reliable audio pause/resume behavior, and a webcam debug overlay that displays webcam status and detected lane in real time. That debug overlay was especially useful for diagnosing gesture mapping problems during testing.

In summary, the implementation phase was not just about writing features, but about engineering reliability. We designed around real-world constraints like camera latency, noisy motion input, and balancing difficulty for different players. By separating systems into scenes and controllers, adding defensive checks, and tuning interaction timing carefully, we transformed a rough prototype into a stable, playable gesture game.

Thank you.
